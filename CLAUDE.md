# Eralands Notifications

REST API на FastAPI для приёма заявок с лендингов. Будущий бот живёт здесь же, в `app/bot/`.

## Стек

- Python 3.12, FastAPI, pydantic-settings
- PostgreSQL 17, SQLAlchemy 2 (async), Alembic, asyncpg
- Менеджер зависимостей: **poetry**
- Линтер/форматтер: ruff
- Запуск: локально (`make run`) или Docker (`make docker-run`)

## Структура

```
app/
  main.py             точка входа (uvicorn)
  core/               фабрика, конфиги, логи, обработчики ошибок, движок БД
  api/                роутеры и зависимости (сейчас /api/health, deps.get_session)
  middleware/         мидлвары (логирование запросов)
  models/             ORM-модели и декларативная база
  schemas/            Pydantic-схемы запросов/ответов
  utils/              утилиты (lifespan)
alembic/              окружение и версии миграций
alembic.ini           конфигурация Alembic
pyproject.toml        зависимости (poetry)
.env.example          шаблон переменных окружения
Dockerfile            uv-based, multi-stage не нужен
docker-compose.yml    сервисы app + postgres
Makefile              команды разработчика
```

## Команды

- `make setup` — копирует `.env`, ставит зависимости (`poetry install`)
- `make run` — поднимает postgres (`db-up --wait`) и запускает сервис локально через `python -m app.main`
- `make db-up` / `make db-down` — поднять/остановить контейнер postgres
- `make format` — `ruff format .` + `ruff check --fix .`
- `make update` — обновляет зависимости через poetry
- `make migration m="..."` — создаёт авто-миграцию по diff моделей
- `make migrate` — применяет миграции до head
- `make test` — поднимает postgres и прогоняет pytest (тестовая БД `notifications_test` создаётся автоматически)

## Конвенции

- Все настройки через переменные окружения с разделителем `__` (например, `APP__HOST`).
  Группа добавляется новым `BaseModel` в `app/core/config.py` и полем в корневом `Settings`.
- Все ошибки уходят через `app/core/errors.py` с унифицированным телом `ErrorResponse`.
- Эндпоинты регистрируются в `app/api/router.py` с префиксом `/api`.
- Модели наследуются от `app.models.base.BaseModel` и регистрируются импортом в `app/models/__init__.py`, иначе их не увидит Alembic autogenerate.
- Docstrings на русском. Технические комментарии в коде минимальны: только когда WHY не очевиден.
- В prod (`APP__ENV=prod`) Swagger/ReDoc отключены.

## Тесты

Тесты этого сервиса пишутся **только как pipeline-сценарии бизнес-логики**. Не «один тест на эндпоинт» и не «один тест на if». Один тест проходит весь путь юзкейса: что сделал админ, что сделал бот, что прилетело с лендинга, что в итоге лежит в БД.

### Что покрывать

Каждый бизнес-кейс — один pipeline-тест в `tests/pipelines/`:

- **Онбординг клиента.** POST `/clients` → POST `/clients/{id}/landings` → бот ввёл код → POST `/leads` → проверили, что `Delivery` создались на каждый активный маршрут.
- **Добавление сотрудника.** Уже есть один маршрут → POST `/landings/{id}/linking-codes` → второй чат привязался → следующий лид породил `Delivery` в оба канала.
- **Уход клиента.** PATCH `/clients/{id}` `is_active=false` → POST `/leads` отдаёт 409 и не создаёт `Delivery` → re-enable → лид снова принимается.
- **Отключение конкретного лендинга.** PATCH `/landings/{id}` `is_active=false` → лид по этому токену 409, лиды по другим токенам того же клиента живут.
- **Отвязка канала.** DELETE `/landings/{id}/routes/{channel_id}` → следующий лид идёт только в оставшиеся каналы; канал и его маршруты на других лендингах не страдают.
- **Линковка.** Невалидный / просроченный / уже использованный код → бот реагирует корректно, состояние БД не меняется.

### Чего НЕ писать

- Тестов на Pydantic-схемы (валидацию полей, обязательность, сериализацию). Это уже проверяет сам Pydantic.
- Тестов на отдельные методы `app/services/db/*` (тонкая обвязка над SQLAlchemy).
- Тестов на доменные сервисы в изоляции от HTTP-уровня. Только в составе пайплайна через клиент.
- Параметризованных тестов с десятками комбинаций входных данных «на всякий случай».
- Тестов на `health`, `errors.py`-форматирование, request-логирование, тривиальные геттеры.

### Инфраструктура

- Реальный Postgres из `make db-up`. Отдельная БД `notifications_test` создаётся автоматически фикстурой `tests/conftest.py`.
- Миграции (`alembic upgrade head`) применяются один раз на сессию pytest. Между тестами — TRUNCATE всех таблиц (транзакционная изоляция не работает: `get_session` коммитит на выходе из запроса).
- Бот в тестах не поднимается, lifespan не запускается, Telegram API не дёргается. Линковка чата проверяется прямым вызовом `LinkingService.from_session(session)`. Это часть того же пайплайна, просто без HTTP-уровня бота.
- HTTP-вызовы — через `httpx.AsyncClient` поверх `ASGITransport`, без uvicorn.
- Админский bearer берётся из `API__KEY`, который conftest выставляет в `test-admin-key` для тестового окружения.

### Если просят добавить тест

Сначала уточнить, какой бизнес-сценарий покрываем целиком. Если ответ «никакой, просто покрытие функции» — такой тест не пишем; вместо этого предлагаем ближайший пайплайн, в который функция уже встроена.
